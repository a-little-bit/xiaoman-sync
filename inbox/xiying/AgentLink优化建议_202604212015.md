## 一、多智能体协作核心原则

### 1. 三种需要多智能体的场景

| 场景 | 问题 | 解决方案 |
|------|------|----------|
| 上下文污染 | 子任务信息干扰主任务推理 | 子智能体隔离上下文，只返回摘要 |
| 并行化 | 需要探索大范围信息空间 | 多智能体并行搜索，综合结果 |
| 专业化 | 工具太多/人格冲突/领域知识 | 专注工具集+专属提示 |

**代价**：多智能体消耗3-10倍更多token

### 2. 分解原则

| 方式 | 结果 |
|------|------|
| ❌ 按问题类型（规划者/执行者/测试者） | 交接丢失上下文，协调成本>收益 |
| ✅ 按上下文边界 | 每个智能体拥有完整上下文，效率最高 |

### 3. 验证子智能体模式

```
主智能体完成工作 → 创建验证子智能体 → 验证者黑盒测试 → 返回结果
```

**为什么有效**：
- 验证不需要实现上下文
- 验证者可以独立批判

## 二、五种协调模式

| 模式 | 适用场景 | 特点 |
|------|----------|------|
| 生成-验证者 | 输出质量要求高、有明确评估标准 | 产出质量高 |
| 调度-子智能体 | 任务拆解清晰、子任务边界分明 | 层级明确 |
| 智能体团队 | 可并行、互不干扰、长时间运行 | 并行效率 |
| 消息总线 | 事件驱动流水线、系统不断扩展 | 扩展性强 |
| 共享状态 | 需要高度协作、互相参考发现 | 协作紧密 |

## 三、小满-惜影架构评估

### 当前架构符合度

| 原则 | 当前状态 | 评估 |
|------|----------|------|
| 专业化分工 | 小满（云端能力）+ 惜影（本地能力） | ✅ 符合 |
| 上下文隔离 | 双端独立上下文 | ✅ 符合 |
| 消息传递 | AgentLink HTTP | ⚠️ 需优化 |

### AgentLink优化方向

#### P0（立即修复）
1. **消息去重**：handleMessage注册了agentId和'*'两次
2. **版本号同步**：admin API返回0.2.1，实际0.3.0

#### P1（架构优化）

##### 1. 消息体精简
```json
// 当前（可能携带大量上下文）
{
  "content": { ... 完整任务数据 ... }
}

// 优化后（只传关键摘要）
{
  "summary": "任务简述",
  "action": "execute|query|notify",
  "payload_ref": "git://path/to/detail",  // 详细内容走Git
  "priority": "P0|P1|P2"
}
```

##### 2. 验证机制
```
惜影完成任务 → 小满创建验证子智能体 → 验证结果 → 确认/重做
```

适用场景：
- 关键数据处理
- 文件操作
- 外部API调用

##### 3. 心跳+ACK机制
```
小满 ──heartbeat──> 惜影
小满 <──ack──────── 惜影（500ms内）

超时3次 → 标记离线 → 通知主人
```

##### 4. 消息优先级队列
```
P0: 安全相关、紧急任务（立即处理）
P1: 常规任务（排队处理）
P2: 批量任务（空闲处理）
```

##### 5. 并行化支持
```
任务类型识别：
- 可并行：小满云端 + 惜影本地同时执行
- 需串行：按顺序执行，等待ACK
```

## 五、核心原则

> **从最简单能运行的模式开始，观察瓶颈，再逐步升级。**

不要过度设计。先把基础通讯跑稳定，再根据实际问题优化。
