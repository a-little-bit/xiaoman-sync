# 奔走夜市项目 - 多角色需求分析体系

> 版本：v1.0
> 创建时间：2026-04-17
> 用途：辅助产品需求分析，确保多角度、全方位审视需求

---

## 一、体系总览

本体系包含三大类角色，共计 **24 个角色**，形成完整的辩证分析网络：

| 类别 | 角色数量 | 核心作用 |
|------|---------|---------|
| A. 利益相关方角色 | 12个 | 代表不同群体的真实诉求 |
| B. 辩证思维角色 | 6个 | 六顶思考帽思维模式 |
| C. 方法论角色 | 6个 | 经典分析框架的人格化 |

---

## 二、角色使用指南

### 2.1 分析流程

```
需求进入 → 利益相关方审视 → 辩证思维分析 → 方法论框架评估
    ↓              ↓                  ↓               ↓
  溯源确认     诉求收集          风险与机会       优先级排序
```

### 2.2 角色切换方式

小满在分析需求时，可以切换到不同角色视角：
- **切换命令示例**：「用摊主视角分析这个需求」「用黑帽思维审视风险」
- **自动触发**：根据需求来源自动匹配相关角色

### 2.3 角色协作关系

```
支持者 ←→ 质疑者     （辩证对抗）
  ↓
利益相关方          （诉求输入）
  ↓
六顶思考帽          （全面分析）
  ↓
方法论角色          （结构化评估）
```

## 三、A类角色：利益相关方角色

### A1. 夜市觅食者（C端用户）

#### 基本信息
- **角色类型**：利益相关方
- **所属群体**：C端用户（消费者）
- **核心诉求**：好吃、好玩、便捷、划算

#### 思维模式
- **看待问题的方式**：从用户体验出发，关注即时满足
- **决策依据**：
  - 口味好不好（主观感受）
  - 价格便不便宜（性价比）
  - 排队时间长不长（效率）
  - 环境卫不卫生（安全）
- **关注的核心指标**：
  - 寻找摊位的时间成本
  - 排队等待时长
  - 菜品质量与价格匹配度
  - 支付便捷性

#### 典型场景
- **场景1**：周五晚上9点，和朋友到夜市觅食，不知道吃什么
  - 诉求：快速找到想吃的、减少选择困难
  - 痛点：摊位太多、信息不对称
  
- **场景2**：看到网红摊位排了50米长队，要不要排？
  - 诉求：想知道值不值得排、是否有替代选项
  - 痛点：怕踩雷、时间成本高

- **场景3**：吃完后想分享给朋友
  - 诉求：方便记录、方便推荐
  - 痛点：没动力、没工具

#### 需求分析方法
- 这个角色会如何分析需求？
  - 「这个功能对我有什么好处？」
  - 「用起来方便吗？」
  - 「值不值这个价？」
- 会提出什么问题？
  - 「为什么要我操作这么多步？」
  - 「比我直接去吃有什么优势？」
  - 「会不会泄露我的信息？」
- 会关注什么细节？
  - 操作流程是否顺畅
  - 界面是否美观易用
  - 响应速度是否够快

#### 与其他角色的关系
| 关系类型 | 相关角色 | 说明 |
|---------|---------|------|
| 利益冲突 | 摊主商家 | 用户想便宜，摊主想赚钱 |
| 合作关系 | 夜市管理员 | 都需要规范有序的夜市环境 |
| 依赖关系 | 兼职配送员 | 需要配送服务才能到家 |
| 信息提供 | 平台运营 | 从平台获取信息 |

### A2. 摊主商家（B端商家）

#### 基本信息
- **角色类型**：利益相关方
- **所属群体**：B端商家（供给侧）
- **核心诉求**：多赚钱、少麻烦、轻松经营

#### 思维模式
- **看待问题的方式**：从经营效率出发，关注成本收益
- **决策依据**：
  - 这个功能能帮我多卖多少？
  - 操作复不复杂？
  - 要不要额外花钱/花时间？
  - 会不会影响我做生意？
- **关注的核心指标**：
  - 日均订单量
  - 客单价
  - 经营时间成本
  - 平台抽成比例

#### 典型场景
- **场景1**：高峰期同时来了5个线上订单
  - 诉求：不怕漏单、不怕搞错顺序
  - 痛点：手忙脚乱、出错率高

- **场景2**：想知道今天哪些菜卖得好
  - 诉求：简单直观、能看到数据
  - 痛点：没有工具、不懂数据分析

- **场景3**：有人投诉说菜里有头发
  - 诉求：能解释清楚、平台能公正处理
  - 痛点：说不清楚、担心被差评

#### 需求分析方法
- 这个角色会如何分析需求？
  - 「能不能帮我多赚钱？」
  - 「操作方不方便？」
  - 「要投入多少？」
- 会提出什么问题？
  - 「要收费吗？收多少？」
  - 「我现在很忙，有时间学吗？」
  - 「万一生意不好怎么办？」
  - 「平台会不会压我的钱？」
- 会关注什么细节？
  - 功能是否简单易学
  - 资金回笼周期
  - 平台规则的公平性
  - 竞争对手有没有这个功能

#### 与其他角色的关系
| 关系类型 | 相关角色 | 说明 |
|---------|---------|------|
| 利益冲突 | 夜市觅食者 | 价格博弈 |
| 合作关系 | 地推BD | BD拓展商户入驻 |
| 依赖关系 | 平台运营 | 依赖平台获取流量 |
| 监督关系 | 夜市管理员 | 需要遵守管理规定 |

### A3. 夜市管理员

#### 基本信息
- **角色类型**：利益相关方
- **所属群体**：管理方（场地方）
- **核心诉求**：管好夜市、安全有序、纠纷少

#### 思维模式
- **看待问题的方式**：从管理效率出发，关注秩序与安全
- **决策依据**：
  - 这个功能会不会让管理更麻烦？
  - 出事了能不能追溯？
  - 商户会不会有意见？
- **关注的核心指标**：
  - 商户合规率
  - 投诉纠纷数量
  - 安全事故数量
  - 管理效率

#### 典型场景
- **场景1**：某摊主和顾客吵起来了
  - 诉求：能快速知道发生了什么、方便调解
  - 痛点：没有证据说不清楚

- **场景2**：夜市结束后场地一片狼藉
  - 诉求：能找到是谁的责任
  - 痛点：没有证据

- **场景3**：有人举报某摊位卖假货
  - 诉求：能核实、能处理
  - 痛点：证据难收集

#### 需求分析方法
- 会提出什么问题？
  - 「出了事能不能查到是谁的问题？」
  - 「商户会不会配合？」
  - 「会不会增加我的工作量？」
- 会关注什么细节？
  - 投诉处理流程
  - 责任界定机制
  - 数据可追溯性

#### 与其他角色的关系
| 关系类型 | 相关角色 | 说明 |
|---------|---------|------|
| 管理关系 | 摊主商家 | 监督、管理摊位 |
| 服务关系 | 夜市觅食者 | 为消费者提供环境 |
| 指导关系 | 平台运营 | 平台规则需要与夜市协调 |

### A4. 地推BD（商务拓展）

#### 基本信息
- **角色类型**：利益相关方
- **所属群体**：平台运营团队
- **核心诉求**：完成拓展任务、拿到提成、建立关系

#### 思维模式
- **看待问题的方式**：从业绩目标出发，关注可执行性
- **决策依据**：
  - 这个摊位好不好谈？
  - 谈成了我能拿多少提成？
  - 需要跑几趟？
- **关注的核心指标**：
  - 入驻摊位数量
  - 洽谈成功率
  - 活跃摊位率

#### 典型场景
- **场景1**：拿着宣传资料去扫街
  - 诉求：话术简单、资料好看、噱头吸引人
  - 痛点：摊位老板不信任、没时间听介绍

- **场景2**：教摊主用App入驻
  - 诉求：操作越简单越好
  - 痛点：摊主文化水平参差不齐

#### 需求分析方法
- 会提出什么问题？
  - 「跟摊主怎么说？」
  - 「摊主最关心什么？」
  - 「入驻流程复杂吗？」
- 会关注什么细节？
  - 物料是否专业
  - 入驻流程是否简便
  - 提成政策是否有吸引力

## 四、B类角色：辩证思维角色（六顶思考帽）

### B1. 白帽 - 客观分析者

#### 基本信息
- **角色类型**：辩证思维
- **来源框架**：六顶思考帽
- **核心诉求**：事实、数据、中立

#### 思维模式
- **如何看待问题**：只关注客观事实，不带感情色彩
- **决策依据**：数据说话、证据支撑
- **关注的核心指标**：
  - 数据的完整性和准确性
  - 统计口径是否一致
  - 信息来源是否可靠

#### 分析方法论
```
提问清单：
□ 这个需求基于什么数据？
□ 数据来源是什么？可靠吗？
□ 有没有遗漏关键信息？
□ 与行业基准相比如何？
□ 历史数据支撑这个判断吗？
```

#### 在夜市项目中的应用
- 分析夜市人流量数据
- 对比竞品功能差异
- 评估市场调研报告
- 验证用户反馈真实性

### B2. 红帽 - 情感直觉者

#### 基本信息
- **角色类型**：辩证思维
- **来源框架**：六顶思考帽
- **核心诉求**：感受、直觉、情绪

#### 思维模式
- **如何看待问题**：从情感和直觉出发，不讲道理
- **决策依据**：内心感受、本能反应
- **关注的核心指标**：
  - 用户情感体验
  - 第一印象
  - 社区氛围

### C1. 第一性原理思考者

#### 基本信息
- **角色类型**：方法论
- **来源框架**：第一性原理
- **核心诉求**：追根溯源、本质思考

#### 思维模式
```
核心问题链：
为什么要做这个？（目的）
一定要这样做吗？（方法）
有没有更本质的解决方案？（本质）
最底层的需求是什么？（核心）
```

#### 分析框架
```
第一性原理分析三问：
1. 我们为什么要做这件事？
   → 剥离表象，找到真正目的

2. 这个需求背后的人性是什么？
   → 马斯洛层次：生理/安全/社交/尊重/自我实现

3. 如果从零开始，会设计成什么样？
   → 打破路径依赖，重新思考
```

#### 在夜市项目中的应用示例
| 需求 | 第一性原理追问 | 本质答案 |
|------|--------------|---------|
| 「加个收藏功能」 | 为什么要收藏？ | 想下次还能吃到/推荐给朋友 |
| 「我要排名功能」 | 为什么想要排名？ | 想知道自己和别人比怎么样 |
| 「加个评价功能」 | 为什么要评价？ | 想帮助别人做选择/想被认可 |

#### 需求分类

```
                    满意度
                      ↑
                 兴奋型需求 ★
                      │
基本型需求 ■ ─────────┼────────→ 实现程度
                      │
                 无差异需求 ○
                      ↓
```

| 类型 | 定义 | 不做会怎样 | 做了会怎样 | 优先级 |
|------|------|-----------|-----------|--------|
| 基本型 | 必须有的功能 | 用户极度不满 | 满意度持平 | P0 |
| 期望型 | 用户期待的功能 | 满意度下降 | 满意度提升 | P1 |
| 兴奋型 | 超预期的功能 | 无影响 | 满意度大幅提升 | P2 |
| 无差异 | 有没有无所谓 | 无影响 | 无影响 | P3 |

#### 在夜市项目中的应用

| 功能 | KANO分类 | 分析依据 |
|------|---------|---------|

| 下单支付 | 基本型 | 核心交易闭环 |
| 菜品评价 | 期望型 | 用户会期待看评价 |
| 排队预约 | 期望型 | 减少等待是好体验 |
| 美食打卡 | 兴奋型 | 超出预期，带来惊喜 |
| 摇一摇抽奖 | 无差异 | 可能用户无感 |

#### 基本信息

- **来源框架**：SWOT分析
- **核心诉求**：全面评估、战略定位

#### SWOT分析维度

```
                    内部能力
                 优势(S)      劣势(W)
              ┌─────────────┬─────────────┐
    机会(O)  │   SO策略    │   WO策略    │
              │  利用优势   │  克服劣势   │
  外部环境    │  抓住机会   │  抓住机会   │
              ├─────────────┼─────────────┤
    威胁(T)  │   ST策略    │   WT策略    │
              │  利用优势   │  克服劣势   │
              │  回避威胁   │  回避威胁   │
              └─────────────┴─────────────┘
```

#### 夜市项目SWOT示例

```
S (优势)
├── 夜市文化天然有话题性和传播性
├── 美食内容受众广、接受度高
├── 低门槛创业，适合切入B端
└── 线下场景有真实感和信任感

W (劣势)
├── 夜市营业时间短，订单集中
├── 摊主数字化程度低
├── 食品安全监管难度大
└── 场地限制，配送困难

O (机会)
├── 夜经济政策支持
├── 年轻人夜生活需求增长
├── 短视频平台带火网红夜市
└── 本地生活服务市场增长

T (威胁)
├── 美团/饿了么可能切入
├── 疫情等不可控因素
├── 消费者对食品安全的担忧
└── 城市管理政策变化
```

### C6. 逆向思维者

#### 基本信息
- **角色类型**：方法论
- **来源框架**：逆向思维
- **核心诉求**：打破常规、换个角度

#### 逆向思维框架

```
正向思维：用户想要XX功能 → 我们做不做？
     ↓
逆向思维：
├── 如果去掉这个功能会怎样？
├── 竞争对手为什么没做？
├── 用户真的需要还是我们以为需要？
├── 不做会有什么替代方案？
└── 做了最坏的结果是什么？
```

#### 逆向问题清单

```
逆向分析三步：
1. 假设这个功能不存在
   → 用户怎么解决？真的解决不了吗？

2. 假设竞争对手做了我们没做
   → 用户会流失吗？流失速度有多快？

3. 假设我们做了但失败了
   → 损失多大？能承受吗？
   → 成功需要什么条件？满足吗？
```

#### 在夜市项目中的应用示例

| 正向需求 | 逆向追问 | 结论 |
|---------|---------|------|
| 「做个抽奖功能」 | 不做抽奖，用户会不会就不来了？ | 不一定，但可能影响活跃度 |
| 「接入外卖配送」 | 不做配送，用户会怎样？ | 可能流失部分懒人用户，但夜市场景可能本来就不需要 |
| 「做个会员体系」 | 用户真的需要会员吗？ | 夜市消费本身是低频场景，会员价值有限 |
