# 夜市平台推荐算法设计方案

> **文档版本**: v1.0  
> **创建日期**: 2026-04-16  
> **核心目标**: 实现"少填偏好、精准推荐"的商户-用户匹配系统

---

## 一、需求背景与设计目标

### 1.1 核心痛点

| 痛点 | 现状 | 期望 |
|------|------|------|
| 用户不愿填写复杂问卷 | 偏好设置步骤繁多 | 零步骤/极简交互 |
| 新用户无历史数据 | 冷启动推荐效果差 | 首次即可获得精准推荐 |
| 用户决策成本高 | 在海量摊位中迷失 | 快速锁定目标选择 |

### 1.2 设计目标

```
┌─────────────────────────────────────────────────────────────────┐
│                        推荐系统核心目标                            │
├─────────────────────────────────────────────────────────────────┤
│  🎯 最少输入：0次主动偏好填写，即得个性化推荐                      │
│  🎯 最快响应：LBS实时定位+时段场景感知                           │
│  🎯 最高精准：行为学习+社交关系+多维度融合                        │
│  🎯 最低门槛：拍照即推荐、语音即搜索                              │
└─────────────────────────────────────────────────────────────────┘
```

## 二、少填偏好的5种核心方案

### 2.1 方案一：位置优先（LBS自动感知）

**原理**: 基于用户实时地理位置，自动推荐周边商户，无需任何手动输入

**技术实现**:
```
┌──────────────────────────────────────────────────────┐
│                   LBS 推荐引擎                        │
├──────────────────────────────────────────────────────┤
│                                                      │
│  GPS定位 ──┐                                         │
│  WiFi定位 ─┼──→ 多源融合定位 ──→ GeoHash网格 ──→ 商户召回 │
│  基站定位 ─┘           │                │           │
│                       ▼                ▼           │
│                 精度校验        地理围栏过滤        │
│                       │                │           │
│                       └───────┬────────┘           │
│                               ▼                     │
│                    距离排序 + 实时状态筛选            │
│                               │                     │
│                               ▼                     │
│                    用户可见：附近100m内摊位列表        │
└──────────────────────────────────────────────────────┘
```

**关键算法**:

| 算法 | 说明 | 权重 |
|------|------|------|
| Haversine距离计算 | 地球表面两点距离公式 | 核心 |
| GeoHash空间索引 | 按网格划分快速检索周边 | 性能优化 |
| 实时排队状态 | 结合摊位实时客流推荐 | +20%优先级 |
| 步行距离估算 | 考虑实际路线而非直线距离 | 场景适配 |

**适用场景**:
- 用户刚进入夜市
- 快速了解周边有什么
- 突发性消费决策

### 2.2 方案二：行为数据学习（隐式反馈）

**原理**: 用户无需主动告知偏好，系统自动从点击、浏览、停留、下单等行为中学习

**隐式反馈信号采集**:

```
行为信号权重表（参考业界标准）:

┌────────────────┬──────────┬────────────────────────────────┐
│ 行为类型        │ 信号强度  │ 说明                           │
├────────────────┼──────────┼────────────────────────────────┤
│ 下单购买        │ +5.0     │ 最强正反馈，明确购买意图        │
│ 加入收藏        │ +4.0     │ 强正反馈，主动标记兴趣          │
│ 完整浏览菜品页   │ +3.0     │ 高兴趣，展示停留时长            │
│ 点击摊位卡片    │ +2.0     │ 中等兴趣，产生点击行为          │
│ 曝光未点击      │ -1.0     │ 轻微负反馈，未引起注意          │
│ 快速划走        │ -2.0     │ 明显负反馈，明确不感兴趣        │
│ 差评/投诉       │ -5.0     │ 最强负反馈                      │
└────────────────┴──────────┴────────────────────────────────┘
```

**行为序列建模**:
```python
# 用户行为序列权重计算示例
user_behavior_weight = {
    "view_detail_page": 3.0,      # 停留>10秒的详情页浏览
    "view_short": 1.5,             # 短暂浏览
    "add_to_cart": 4.0,           # 加购
    "purchase": 5.0,              # 购买
    "collect": 4.0,               # 收藏
    "negative_swipe": -2.0,       # 快速滑走
    "no_action_after_exposure": 0  # 曝光无行为(需结合上下文)
}
```

**多行为融合策略**:
- **多行为对齐(MBA)**: 同一用户的点击、购买、收藏等行为应反映相似偏好
- **去噪处理**: 识别误点击、冲动购买等噪声数据
- **时间衰减**: 近期行为权重高于历史行为

**适用场景**:
- 复购用户再次访问
- 个性化推荐列表
- 偏好标签自动生成

### 2.3 方案三：社交关系推荐（朋友去过的地方）

**原理**: 基于用户社交网络，推荐朋友去过/好评的摊位

**社交推荐链路**:

```
┌───────────────────────────────────────────────────────────┐
│                    社交关系图谱                             │
├───────────────────────────────────────────────────────────┤
│                                                           │
│   用户A ──[好友]──→ 用户B ──[去过]──→ 摊位X              │
│      │                  │                                 │
│      │                  └──[好评]──→ 推荐权重 +30%         │
│      │                                                       │
│      └──[同社区]──→ 用户C ──[收藏]──→ 摊位Y               │
│                                                           │
│   推荐逻辑: 朋友去过+朋友好评 → 强烈推荐                    │
└───────────────────────────────────────────────────────────┘
```

**社交信号类型**:

| 社交行为 | 推荐权重加成 | 说明 |
|----------|-------------|------|
| 好友购买 | +50% | 最强社交信号 |
| 好友收藏 | +40% | 强兴趣表达 |
| 好友好评(4-5星) | +35% | 口碑背书 |
| 好友浏览(同品类) | +20% | 潜在兴趣 |
| 多人同时在摊 | +25% | 热度信号 |

**隐私保护机制**:
- 仅展示"你朋友圈X人去过"而非具体名单
- 尊重用户隐私设置
- 提供"隐藏我的动态"选项

**适用场景**:
- 首次访问某个夜市
- 信任决策(不知选哪家)
- 社交聚会场景

### 2.4 方案四：时段场景推荐（自动判断意图）

**原理**: 根据当前时间段、天气、节假日等上下文，自动判断用户需求

**时段意图识别表**:

```
┌────────────┬─────────────────────┬────────────────────────────┐
│ 时间段      │ 推测意图             │ 推荐策略                   │
├────────────┼─────────────────────┼────────────────────────────┤
│ 06:00-09:00 │ 早餐/早点           │ 包子、油条、豆浆等         │
│ 11:00-13:30│ 午餐/工作餐          │ 快餐、面食、套餐           │
│ 17:00-19:00│ 下班充饥            │ 小吃、奶茶、煎饼            │
│ 19:00-21:00│ 晚餐/逛吃           │ 烧烤、火锅、小龙虾          │
│ 21:00-24:00│ 夜宵/社交           │ 烤串、啤酒、卤味           │
└────────────┴─────────────────────┴────────────────────────────┘
```

**场景维度融合**:

| 场景因素 | 识别方式 | 推荐影响 |
|---------|---------|---------|
| 工作日 vs 周末 | 系统时间 | 周末推荐家庭/朋友聚餐场景 |
| 天气恶劣(雨/雪) | 气象API | 优先推荐室内/有棚摊位 |
| 气温高低 | 气象API | 热天推荐冷饮，寒天推荐热食 |
| 节假日 | 日历数据 | 推荐节日限定/聚餐套餐 |
| 用户状态(空腹/已吃) | 历史数据+时段 | 已吃推荐解腻，空腹推荐主食 |

**多模态场景感知**:
```
推荐请求 = f(当前位置, 当前时段, 天气状况, 历史行为, 社交热度)
```

**适用场景**:
- 用户打开App无明确目标
- 快速推荐"这个点该吃啥"
- 天气变化时的即时需求

### 3.2 多维度评分公式

**综合评分公式**:

```
Score(用户u, 商户i) = 
    α × Score_LBS(u, i)        # 位置匹配分
  + β × Score_Behavior(u, i)   # 行为偏好分
  + γ × Score_Context(u, i)    # 场景匹配分
  + δ × Score_Social(u, i)     # 社交推荐分
  + ε × Score_Quality(i)       # 商户质量分

其中: α + β + γ + δ + ε = 1
```

**各维度计算**:

| 维度 | 计算方式 | 权重范围 |
|------|---------|---------|
| Score_LBS | 1 / (1 + distance), 距离越近分越高 | 0.15-0.25 |
| Score_Behavior | 协同过滤/CF分数 | 0.25-0.35 |
| Score_Context | 时段×天气×状态匹配度 | 0.10-0.20 |
| Score_Social | 好友关系加权 | 0.10-0.15 |
| Score_Quality | 好评率/复购率/服务分 | 0.15-0.25 |

## 四、冷启动策略

### 4.1 冷启动分类

```
┌─────────────────────────────────────────────────────────────┐
│                     冷启动类型                               │
├─────────────────┬───────────────────┬─────────────────────┤
│   用户冷启动     │   商户冷启动        │    系统冷启动        │
├─────────────────┼───────────────────┼─────────────────────┤
│ 新用户无历史数据 │ 新摊位无订单/评价   │ 平台刚上线           │
│ 首次访问         │ 新品上架           │ 数据量极少           │
│ 匿名/游客用户    │ 跨区域扩张         │ 无用户行为积累       │
└─────────────────┴───────────────────┴─────────────────────┘
```

### 4.2 新用户冷启动策略

**分级应对方案**:

```
┌─────────────────────────────────────────────────────────────────┐
│                     新用户冷启动流程                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Step 1: 0交互 → 基于LBS+时段直接推荐                           │
│          │                                                      │
│          ├─ 获取实时位置 → 计算附近摊位                          │
│          ├─ 获取当前时段 → 推测饮食意图                          │
│          └─ 输出: 基于位置+时段的基础推荐                         │
│                                                                 │
│  Step 2: 首次点击 → 快速学习用户偏好                            │
│          │                                                      │
│          ├─ 用户点击摊位 → 标记品类偏好                          │
│          ├─ 用户浏览详情 → 记录停留时长                          │
│          └─ 输出: 单次行为即可更新推荐模型                        │
│                                                                 │
│  Step 3: 3-5次交互 → 形成初步画像                              │
│          │                                                      │
│          ├─ 品类偏好标签(辣/甜/主食/小吃)                        │
│          ├─ 价格敏感度(人均水平)                                 │
│          └─ 距离偏好(愿意走多远)                                 │
│                                                                 │
│  Step 4: 10+次交互 → 个性化推荐成熟                             │
│          │                                                      │
│          └─ 进入完整协同过滤推荐阶段                              │
└─────────────────────────────────────────────────────────────────┘
```

**LBS首触策略**:
```
新用户推荐权重公式:

Score_new_user = 
    0.4 × Score_LBS           # 位置权重最高
  + 0.3 × Score_Popularity    # 热门/高评分兜底
  + 0.2 × Score_Context       # 时段匹配
  + 0.1 × Score_TimeNew      # 新摊位探索分
```

**探索-利用平衡(ε-greedy)**:
- ε=0.2: 80%时间展示已知偏好
- ε=0.2: 20%时间探索新摊位
- 防止信息茧房，发现用户潜在兴趣

### 5.2 用户意图识别

**意图分类**:

| 意图类型 | 触发条件 | 推荐策略 |
|---------|---------|---------|
| 无明确目的 | 打开App无搜索/点击 | 场景推荐+热门推荐 |
| 品类意图 | 搜索"烧烤"/"奶茶" |品类排序+距离 |
| 位置意图 | 点击"看看附近" | LBS优先+实时状态 |
| 特定商户 | 搜索具体摊位名 | 直接展示+关联推荐 |
| 社交意图 | 点击"朋友推荐" | 社交关系过滤 |
| 比价意图 | 对比多个摊位 | 提供差异化信息 |

### 5.3 商户曝光分配

**流量分配模型**:

```
每日曝光预算 = Base_Exposure × Quality_Score × Activity_Level

其中:
- Base_Exposure: 基础曝光量(根据摊位等级)
- Quality_Score: 质量分(评分、好评率、服务分)
- Activity_Level: 活跃度(当日订单量、在线时长)
```

**曝光调控规则**:

| 商户状态 | 曝光策略 |
|---------|---------|
| 高质量+高活跃 | 正常曝光，优先展示 |
| 高质量+低活跃 | 推送提醒，提升在线 |
| 低质量+高活跃 | 监控投诉，控制曝光 |
| 低质量+低活跃 | 减少曝光，整改提示 |
| 新上线 | 冷启动流量池 |
| 违规处罚中 | 限制曝光 |
