## 核心观点

**单智能体系统足以处理大多数企业工作流，多智能体架构仅在特定场景下提供额外价值。**

---

## 什么时候需要多智能体？

Anthropic团队发现三种场景下多智能体明显优于单智能体：

### 1. 上下文污染（Context Protection）

**问题**：大模型上下文窗口有限，当一个子任务的信息对后续子任务无关时，上下文污染导致性能下降。

**例子**：客服智能体需要同时处理订单历史和技术问题诊断
- 单智能体：订单查询添加2000+ tokens到上下文，影响技术问题推理
- 多智能体：订单查询智能体处理历史，只返回50-100 tokens摘要给主智能体

**适用场景**：
- 子任务产生大量上下文（>1000 tokens）但大部分信息与主任务无关
- 子任务边界清晰，有明确的提取标准
- 检索/查找类操作

### 2. 并行化（Parallelization）

**优势**：多个智能体并行运行可以探索更大的搜索空间

**例子**：Anthropic的Research功能
- 主智能体分解查询为多个方面
- 子智能体并行调查不同方面
- 主智能体综合结果

**代价**：多智能体系统通常消耗**3-10倍**更多tokens
- 每个智能体需要自己的上下文
- 智能体之间需要交换消息协调
- 结果传递时需要总结

**核心收益**：更全面的结果，而非更快的速度

### 3. 专业化（Specialization）

**适用场景**：
- **工具集专业化**：智能体有20+工具时选择困难
- **系统提示专业化**：不同任务需要冲突的人格/约束
- **领域知识专业化**：法律分析、医学研究需要深度领域上下文

**例子**：多平台集成系统
- CRM智能体：8-10个CRM工具 + 专业提示
- 营销智能体：8-10个营销工具 + 专业提示
- 编排智能体：路由请求到合适的专家

## 决策框架

### 从单智能体开始

多智能体引入开销：
- 每个智能体是潜在的故障点
- 需要维护更多提示
- 可能产生意外行为

**建议**：从最简单的能运行的模式开始，观察瓶颈，再逐步升级

### 判断标准

| 信号 | 说明 |
|------|------|
| 接近上下文限制 | 智能体常规使用大量上下文，性能下降 |
| 管理大量工具 | 智能体有15-20+工具，选择困难 |
| 可并行子任务 | 任务自然分解为独立部分 |

**注意**：Tool Search Tool可以减少85%的token使用，提高工具选择准确性

## 上下文中心分解原则

**关键洞察**：按照上下文边界分解工作，而非问题类型

### 问题中心分解（通常适得其反）
- 按工作类型分工（规划者/执行者/测试者/审核者）
- 每次交接丢失上下文
- 协调成本超过收益

### 上下文中心分解（通常有效）
- 按上下文边界分工
- 一个智能体处理一个功能及其测试，因为它已有必要上下文
- 仅在上下文真正隔离时拆分

**有效分解边界**：
- 独立研究路径（亚洲市场 vs 欧洲市场）
- 有清晰接口的独立组件
- 黑盒验证

**问题分解边界**：
- 同一工作的顺序阶段
- 紧密耦合的组件
- 需要共享状态的工作

## 验证子智能体模式

**一个跨领域持续有效的模式**：验证子智能体

### 原理
主智能体完成工作 → 创建验证子智能体 → 验证者独立测试 → 返回结果

### 优势
- 验证本质不需要上下文传递
- 验证者可以黑盒测试，不需要了解实现历史
- 保留完整的批判能力

### 适用场景
- 质量保证：运行测试套件、代码检查
- 合规检查：验证文档符合政策要求
- 输出验证：确认生成内容符合规格
- 事实验证：单独智能体验证声明或引用

### 常见失败模式：早胜利问题
验证者运行1-2个测试就标记通过

**缓解策略**：
- 明确标准："运行完整测试套件并报告所有失败"
- 全面检查：测试多个场景和边缘情况
- 负面测试：尝试应该失败的输入，确认确实失败
- 明确指令："必须在标记通过之前运行完整测试套件"

## 实施建议

### 三步确认

1. **确认存在真正的约束**
   - 上下文限制、并行化机会、专业化需求
   
2. **分解遵循上下文，而非问题类型**
   - 按所需上下文分组工作
   
3. **存在清晰的验证点**
   - 子智能体可以在不需要完整上下文的情况下验证工作

### 核心原则

> **从最简单可行的方法开始，仅在证据支持时添加复杂性。**

## 五种协调模式

参考飞书文档翻译的多智能体协作指南：

| 模式 | 适用场景 | 特点 |
|------|----------|------|
| 生成-验证者 | 看重输出质量、有明确评估标准 | 产出质量高 |
| 调度-子智能体 | 任务拆解清晰、子任务边界分明 | 层级明确 |
| 智能体团队 | 可并行处理、互不干扰、长时间运行 | 并行效率 |
| 消息总线 | 事件驱动流水线、系统不断扩展新智能体 | 扩展性强 |
| 共享状态 | 需要高度协作、智能体之间互相参考发现 | 协作紧密 |

## 对AgentLink的启示

### 当前架构评估

AgentLink目前采用 **消息总线 + 点对点通讯** 模式：
- 小满（扣子云端）↔ 惜影（本地服务器）
- HTTP API通讯
- 支持消息处理和路由

### 优化方向

1. **上下文隔离**
   - 小满和惜影各自维护独立上下文
   - 消息传递只传递必要的摘要信息

2. **任务专业化**
   - 小满：云端能力（扣子生态、飞书同步、AI资讯推送）
   - 惜影：本地能力（数据库、文件系统、长期运行任务）

3. **验证机制**
   - 关键任务完成后，可创建验证子智能体
   - 独立验证结果质量

4. **并行化机会**
   - 多个任务可以并行执行（小满云端 + 惜影本地）
   - 例如：小满推送资讯同时惜影处理数据分析

### 下一步优化建议

1. **消息持久化**：记录所有通讯历史，支持追溯
2. **ACK机制**：确保消息可靠送达
3. **心跳检测**：及时发现对方离线
4. **任务队列**：支持任务优先级和排队
