### 2.2 架构模式

| 模式 | 应用场景 | 实现方式 |
|------|---------|---------|
| **微服务架构** | 服务解耦、独立扩展 | Spring Cloud / Go / gRPC |
| **读写分离** | 数据库读多写少 | 主从复制、只读副本 |
| **分库分表** | 海量数据存储 | MySQL Sharding / PostgreSQL Citus |
| **缓存分层** | 热点数据加速 | L1本地缓存 + L2 Redis |
| **消息队列异步** | 流量削峰、服务解耦 | Kafka / RocketMQ |
| **服务网格** | 服务治理、流量管理 | Istio / Envoy |

### 3.2 技术栈选型理由

#### 服务框架选型
- **Spring Cloud Alibaba**: 阿里开源，与阿里云深度集成，支持Nacos、Sentinel、Dubbo
- **Go + gRPC**: 高并发场景首选，内存占用低，适合I/O密集型服务

#### API网关选型
- **APISIX**: 基于Apache顶级项目，性能优秀（nginx + Lua + OpenResty），支持多语言插件
- **Kong**: 成熟度高，插件生态丰富，Konga管理界面友好

#### 数据库选型
- **PostgreSQL + Citus**: 
  - Citus扩展实现水平分片，分布式SQL处理
  - PostGIS支持地理位置查询
  - 强一致性ACID保证
- **TiDB (备选)**: 水平扩展、MySQL兼容，适合强一致性OLTP场景

**优化策略：**

| 策略 | 实现方式 | 效果 |
|------|---------|------|
| CDN加速 | 静态资源(图片/CSS/JS)全量上CDN | 降低源站压力80%+ |
| 多地域部署 | 华东/华北/华南/海外多机房 | 就近访问、容灾备份 |
| DNS智能解析 | 地域/运营商/负载智能调度 | 访问延迟降低50%+ |
| HTTPS/HTTP2 | SSL卸载、HTTP/2多路复用 | 提升连接效率 |
| HTTP/3(QUIC) | UDP协议、0-RTT握手 | 弱网场景优化 |
| 证书自动化 | Let's Encrypt + 自动续期 | 证书管理零运维 |

### 4.2 网关层设计

```yaml
# API网关核心配置
gateway:
 限流策略:
   - 全局限流: 100万QPS/全局
   - 用户级: 1000 QPS/用户
   - 接口级: 按接口动态配置
   - 热点限流: 基于参数值的令牌桶
   算法: 令牌桶 + 滑动窗口组合

 熔断策略:
   - 慢调用比例: >30% 持续5秒触发
   - 异常比例: >50% 立即触发
   - 异常数: >100次/10秒触发
   - 熔断恢复: 半开试探 → 逐步恢复

 降级策略:
   - 核心功能保底: 用户信息/关键业务流程
   - 非核心降级: 推荐/广告/扩展功能
   - 静态化降级: 热点数据降级到CDN
   - 读降级: 从缓存/从库读取
```

### 4.3 服务层设计

#### 无状态服务设计

```yaml
# 服务无状态化原则
stateless:
 - 会话存储: Redis集群，不落本地内存
 - 配置中心: Nacos/Apollo动态配置
 - 本地缓存: 仅做L1缓存，TTL<60秒
 - 定时任务: 分布式调度(XLARK-Job)
 - 扩展策略: Horizontal Pod Autoscaler (HPA)
```

#### 服务隔离策略

| 隔离级别 | 场景 | 实现方式 |
|---------|------|---------|
| **线程隔离** | 不同接口组 | 线程池隔离 |
| **进程隔离** | 核心/非核心服务 | K8s Pod独立部署 |
| **集群隔离** | 高优先级用户 | 独立服务集群 |
| **机房隔离** | 容灾需求 | 多机房独立部署 |

### 4.4 数据层设计

#### 分库分表策略

```
┌──────────────────────────────────────────────────────────────┐
│                     PostgreSQL + Citus 分片架构               │
├──────────────────────────────────────────────────────────────┤
│  分片键选择:                                                   │
│  - 用户表(user): user_id % 64                                 │
│  - 消息表(message): (user_id + receiver_id) % 256             │
│  - 社交关系表(relation): user_id % 32                         │
├──────────────────────────────────────────────────────────────┤
│  分片数规划:                                                   │
│  - 阶段1(100万用户): 8分片                                    │
│  - 阶段2(1000万用户): 64分片                                  │
│  - 阶段3(1亿用户): 256分片                                    │
├──────────────────────────────────────────────────────────────┤
│  扩容策略:                                                     │
│  - 在线扩容: Citus自动重平衡                                  │
│  - 双写方案: 旧库+新库同时写入                                │
│  - 数据归档: 冷热分离，历史数据归档                           │
└──────────────────────────────────────────────────────────────┘
```

#### 缓存穿透/击穿/雪崩防护

| 问题 | 防护策略 | 实现方式 |
|------|---------|---------|
| **缓存穿透** | 空值缓存 + 布隆过滤器 | 缓存NULL值TTL=60s，布隆过滤非法请求 |
| **缓存击穿** | 互斥锁 + 热点数据永不过期 | SETNX加锁，热点数据设置-1 TTL |
| **缓存雪崩** | 多级TTL + 随机偏移 + Redis Cluster | TTL随机±10%，主从+哨兵高可用 |

#### 数据一致性方案

```
强一致性场景 (账户/订单):
  └─> 分布式事务: Seata AT/TCC模式
      └─> 两阶段提交保证ACID

最终一致性场景 (Feed/消息):
  └─> 消息队列异步同步
      └─> RocketMQ事务消息
      └─> 本地消息表 + 定时补偿

缓存一致性:
  └─> Cache Aside模式
      └─> 更新DB → 删除缓存(而非更新)
      └─> 延迟双删保证最终一致
```

**连接池优化：**

| 优化点 | 策略 | 效果 |
|--------|------|------|

### 6.2 群消息广播策略

```yaml
# 分层广播策略
broadcast:
  在线用户数 < 100:
    - 策略: 广播推送
    - 方式: 单次遍历推送

  在线用户数 100-500:
    - 策略: 分批推送
    - 方式: 50用户/批次，间隔10ms

  在线用户数 500-2000:
    - 策略: 环形广播
    - 方式: 层级树状推送，每层并行

  在线用户数 > 2000:
    - 策略: 消息扩散+拉取混合
    - 方式: 
      1. 写入消息队列
      2. 通知在线用户有新消息
      3. 用户主动拉取消息列表
      4. 按需拉取消息详情
```

|---------|------|------|
| 多级缓存 | L1本地+L2 Redis | 命中率>90% |
| 热点区域预加载 | 热门区域数据预加载 | 响应时间<50ms |
| 客户端计算 | 附近排序在客户端 | 减少服务端压力 |
| 降级策略 | 服务繁忙时返回缓存 | 保证可用性 |
| 异步写入 | 位置更新异步入库 | 降低主链路压力 |

## 八、成本优化（OPC模式）

### 8.1 分阶段成本规划

| 阶段 | 时间周期 | 用户规模 | DAU | 架构方案 | 月成本 | 单用户成本 |

| **验证期** | 3-6月 | 10万 | 1万 | 小集群+读写分离 | ¥3,000 | ¥0.3 |
| **增长期** | 6-12月 | 100万 | 10万 | 分布式+CDN | ¥20,000 | ¥0.2 |
| **规模期** | 12-24月 | 1000万 | 100万 | 多机房+高可用 | ¥200,000 | ¥0.2 |
| **成熟期** | 24月+ | 1亿+ | 1000万 | 全球化+智能调度 | ¥1,500,000 | ¥0.15 |

### 8.2 OPC精益原则

```
成本优化核心策略:
┌─────────────────────────────────────────────────────────────────┐
│  1. 按需扩展 (Scale on Demand)                                  │
│     - 自动扩缩容，闲时缩容                                       │
│     - 使用Spot/抢占式实例节省70%成本                            │
│                                                                  │
│  2. 资源复用 (Resource Sharing)                                  │
│     - 混合部署，非核心服务共用资源                               │
│     - 计算与存储分离，按需配置                                   │
│                                                                  │
│  3. 开源优先 (Open Source First)                                 │
│     - 自建MySQL/Redis替代云数据库                                │
│     - MinIO替代对象存储                                         │
│                                                                  │
│  4. 边缘计算 (Edge Computing)                                   │
│     - CDN缓存减少回源                                           │
│     - 边缘计算处理简单请求                                       │
│                                                                  │
│  5. 成本可视化 (Cost Visibility)                                 │
│     - 每日成本报表                                               │
│     - 资源利用率监控                                             │
│     - 异常消费告警                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 8.3 成本占比预估

|-------|------|---------|
| 云服务器/容器 | 40% | Spot实例+混合部署 |
| 数据库 | 20% | 读写分离+缓存优化 |
| CDN/带宽 | 15% | 压缩+缓存优化 |
| 存储 | 10% | 冷热分层+压缩 |
| 监控/日志 | 5% | 开源方案 |
| 其他(安全/维护) | 10% | 自动化运维 |
